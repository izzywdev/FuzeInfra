FuzeOne — Templates × Environments → Sessions

The four engineer roles are reusable, repo-agnostic agent templates (one /v1/agents each). A repo isn't a different agent — it's a different environment bound at launch (toolchain, checkout, networking, creds) plus its repo-expert. A session is one role × environment instantiation; you run many in parallel per feature.

Reading L→R: pick a role template → bind a per-repo environment (this harnesses it to the repo boundary) → get sessions (many, parallel). Cross-product features (billing, RAG, security, identity) are skills/MCP attached at launch (via agent_with_overrides), not per-repo agents. So "fuzefront-backend" = the Backend template × the FuzeFront env (+ its skills/vault) — assembled at sessions.create, not a 16th stored agent.
